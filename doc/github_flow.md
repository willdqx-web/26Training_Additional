# GitHub Flow ガイド

> このドキュメントはプロジェクト全体を通して参照する共通ガイドです。
> 各 day ドキュメントの「GitHub チェックポイント」と合わせて読んでください。

---

## このガイドを読む前に：Git と GitHub の全体像

### Git とは

「ファイルの変更履歴を記録するツール」。誰がいつ何を変えたかを記録し、過去の状態に戻せる。

```
変更A → 変更B → 変更C（現在）
              ↑
           バグが入ったことに気づく → 変更B に戻れる
```

### GitHub とは

「Git の履歴をインターネット上に保存・共有するサービス」。チームが同じコードで作業できる場所。

```
自分の PC（ローカル）          インターネット（GitHub）
  ├── main ブランチ  ←push→   ├── main ブランチ
  └── feature/xxx  ←push→   └── feature/xxx ブランチ
```

### GitHub Flow とは

「GitHub を使ったチーム開発の手順（フロー）」。このプロジェクトで使うルール：

```
1. main から feature ブランチを切る
2. feature ブランチでコードを書いてコミット
3. feature ブランチを GitHub に push する
4. PR（プルリクエスト）を作成して差分を確認
5. 自己レビュー → マージ
6. ブランチを削除
```

このサイクルを機能ごとに繰り返す。

## 1. コミットタイミング

### 原則：1コミット = 1つの論理的な変更

コミットは「作業の記録」ではなく「変更の意味の単位」。

```
❌ 悪い例（作業ログ的なコミット）
  - "作業中"
  - "今日の進捗"
  - "モデル、ビュー、テンプレート、URL全部追加"

✅ 良い例（意味の単位でのコミット）
  - "feat: Taskモデルを追加"
  - "feat: タスク一覧ビューを実装"
  - "fix: ログインしていないユーザーが一覧にアクセスできる問題を修正"
```

### コミットすべきタイミングの判断基準

| タイミング | 理由 |
|-----------|------|
| ✅ 1つの機能が**ブラウザで動作確認**できたとき | 「動く状態」でコミットすることで、そのコミットへの巻き戻しが安全になる |
| ✅ テストが通ったとき | 壊れていない状態を履歴として残せる |
| ✅ 次の作業で今の変更を壊す可能性があるとき | 安全な退避ポイントを作る |
| ❌ 「なんとなく保存したい」だけのとき | コミット粒度が細かくなりすぎ、履歴が読めなくなる |
| ❌ 一日の終わりに全部まとめてコミット | 変更が大きすぎて何をしたか分からなくなる |

### なぜコミット粒度が重要なのか

**細かすぎると**：
- `git log` を見ても「何が変わったか」が分からない
- レビューするとき「この PR で何を達成したのか」が追えない

**大きすぎると**：
- バグが混入したとき `git bisect` でどのコミットが原因か特定できない
- 一部を `git revert` で戻したくても、余分な変更まで戻ってしまう

**適切な粒度**：`git log --oneline` を見たときに、変更の流れがストーリーとして読めること。

## 2. コミットメッセージ規約

### フォーマット

```
type: 変更内容の要約（日本語、50文字以内）
```

### type 一覧と使い分け

| type | 使う場面 | 例 |
|------|---------|-----|
| `feat` | 新機能の追加 | `feat: タスク一覧ビューを実装` |
| `fix` | バグ修正 | `fix: ログイン後のリダイレクト先を修正` |
| `docs` | ドキュメントのみの変更 | `docs: READMEに起動手順を追加` |
| `refactor` | 動作を変えないコードの整理 | `refactor: タスクフィルタ処理をメソッドに切り出し` |
| `test` | テストの追加・修正 | `test: Taskモデルのユニットテストを追加` |
| `chore` | ビルド・設定ファイルの変更 | `chore: requirements.txtにflake8を追加` |
| `style` | コードフォーマットのみ | `style: flake8警告を解消` |

### 良い例 vs 悪い例

```bash
# ❌ 悪い：何を変えたか分からない
git commit -m "修正"
git commit -m "変更"
git commit -m "update"

# ❌ 悪い：長すぎる・動詞が不明確
git commit -m "feat: タスクの作成と編集と削除の機能を全部実装してURLも設定した"

# ✅ 良い：変更内容が一目で分かる
git commit -m "feat: Taskモデルとマイグレーションを追加"
git commit -m "feat: タスク作成・編集・削除ビューを実装"
git commit -m "fix: 他ユーザーのタスクが参照できる脆弱性を修正"
```

### 50文字ルールの理由

`git log --oneline` や GitHub のコミット一覧では1行で表示される。50文字を超えると切れて読めなくなる。

## 3. ブランチ戦略

### main ブランチの原則

`main` は**常にデプロイ可能な状態**を保つ。

> **なぜ**：`main` が壊れていると、そこからブランチを切った他の機能開発も全員が壊れた状態で作業することになる。チーム開発では致命的。

### ブランチの種類と命名

| ブランチ名 | 用途 | 例 |
|-----------|------|-----|
| `feature/xxx` | 新機能開発 | `feature/user-auth` |
| `fix/xxx` | バグ修正 | `fix/login-redirect` |
| `docs/xxx` | ドキュメント | `docs/readme` |
| `chore/xxx` | 設定・環境 | `chore/docker-setup` |

### なぜ機能ごとにブランチを切るのか

**理由①：main の安全を守るため**
作業中の未完成コードが main に入ると、そこからブランチを切った人が壊れたコードを受け取る。

**理由②：変更の独立性を保つため**
「認証機能」と「タスク機能」を同じブランチで作ると、片方が壊れたとき切り離せない。

**理由③：レビューしやすくするため**
1ブランチ = 1機能なら、PR の差分が「この機能だけ」になり、変更意図が明確。

### ブランチを切るタイミング

```bash
# 新しい作業を始める前に必ずブランチを切る
git checkout main           # main の最新状態から切る
git pull origin main        # リモートの最新を取得
git checkout -b feature/xxx # 新しいブランチを作成
```

**なぜ `main` から切るのか**：古い状態から切ると、すでに main にマージされた他機能の変更が含まれず、後でコンフリクトが起きやすくなる。

### ブランチの寿命

```
feature/xxx ブランチを切る
     ↓
実装・コミット
     ↓
PR を作成（動作確認済み）
     ↓
自己レビュー・マージ
     ↓
ブランチを削除（リモート・ローカル両方）
```

ブランチは**PR マージ後に即削除**する習慣をつける。放置すると「どれが作業中か」が分からなくなる。

## 4. PR の書き方

### PR タイトル

コミットメッセージと同じ形式。

```
feat: タスク一覧・詳細ビューを実装
```

### PR 本文テンプレート

```markdown
## 概要
（この PR で何を実装・修正したか、1〜3行で説明）

## 変更内容
- （変更点を箇条書き）
- 例：`ListView` でタスク一覧を実装
- 例：`LoginRequiredMixin` でアクセス制御を追加

## 確認方法
1. （動作確認の手順を番号つきで）
2. 例：`docker compose up` で起動
3. 例：`http://localhost:8000/tasks/` を開く
4. 例：ログイン済みユーザーでタスク一覧が表示されることを確認

## スクリーンショット（任意）
（ブラウザの動作画面を貼るとレビューしやすい）
```

### 自己レビューの手順

PR を作成したら、**マージ前に必ず GitHub 上で差分を確認**する。

1. GitHub の PR ページを開く
2. **Files changed** タブをクリック
3. 差分を1行ずつ確認する
   - 意図していない変更が混入していないか
   - デバッグ用の `print()` が残っていないか
   - `.env` などのシークレットが含まれていないか
4. 気づいたことを**コメントとして残す**（自分へのメモでよい）

## 5. マージのタイミングと方法

### マージ可能な条件

以下をすべて満たしたときにマージする。

- [ ] 実装した機能がブラウザで動作確認できている
- [ ] 自己レビューが完了している（Files Changed タブで確認済み）
- [ ] PR のコメントに少なくとも1件の確認メモがある

### なぜ「動作確認後」にマージするのか

**理由①：main を壊さないため**
未確認コードを main に入れると、次にブランチを切った人が壊れた状態から始まる。

**理由②：後から原因を特定しやすくするため**
「この PR のマージ前後でバグが出た」と言えるためには、PR 単位で「動いている状態」が確定している必要がある。

**理由③：PR の粒度を保つため**
中途半端な状態でマージを繰り返すと、「何が完成していて何が途中か」が履歴から追えなくなる。

### マージ方法：Squash merge を推奨

| マージ方法 | 説明 | 向いている場面 |
|-----------|------|--------------|
| **Merge commit** | ブランチの全コミットが main に残る | 全コミット履歴を残したい場合 |
| **Squash merge** | ブランチの全コミットを1つにまとめてマージ | WIP コミットが多く、履歴をきれいに保ちたい場合 |
| Rebase merge | コミットを1つずつ main に追加 | コミット履歴を線形に保ちたい場合 |

**本プロジェクトでは Squash merge 推奨**：1人開発で WIP コミットが混じりやすいため、PR 単位で履歴をまとめる方が `git log` が読みやすい。

### マージ後の後処理

```bash
# リモートブランチは GitHub の "Delete branch" ボタンで削除
# ローカルブランチも削除する
git checkout main
git pull origin main          # マージされた変更を取得
git branch -d feature/xxx     # ローカルブランチを削除
```
